The north star is the closed clinical loop **Schedule → Plan → Ready → Conduct → Review**,
delivered across four tiers: cloud control plane ↔ in-room Matrix Shell ↔ sandboxed apps ↔
devices. See `MATRIX-ARCHITECTURE.md` for the target design, `MATRIX-INTEGRATION-ADAPTERS.md`

end-to-end; every beat maps to a component + proof, and a beat that stops being demonstrable
is a regression.

## 1. Headline

**Matrix Planner is still vision-complete for its tier, and P1 is now materially closed in
the Shell.** The Planner publishes `matrix.snapshot/2`; `matrix-shell` now pulls
`/api/rooms/:roomId/published-plan`, adapts the envelope payload through
`planner-snapshot-adapter.js`, feeds the existing room/runtime plan pipeline, builds the display
control model from the same snapshot, and has end-to-end shell tests proving the adapter output
drives a case workspace plan.

The **Review / learning loop** also moved forward: the Planner still ingests
`matrix.caseEvent/1` and aggregates it into per-card re-tuning signals, and the Shell now has a
case-event client wired through `createCaseWorkspace(... onTransition ...)` to post lifecycle and
utilization events back to the Planner. This is code-real and test-covered, but still needs a
two-service live smoke against a running Planner/Shell pair with room auth and URLs configured.

The world-class gap has shifted from "build the P1 seam" to **production hardening and platform
services**: real OIDC/JWKS, service/room identity for M2M event ingest, central audit wiring,
Runtime Services, Fleet Registry, real HL7/FHIR/DICOM adapters, and deployment packaging.

## 6. Honest caveats

- The Review loop is **code-wired from Shell to Planner**, but still needs a recorded live
  two-service smoke and production retry/queue behavior.
- `publishedBy` provenance and Planner auth are **dev-grade** until real OIDC/JWKS lands.
- Planner `POST /case-events` is intentionally exempt from auth for the current M2M slice; it
  must move behind service/room identity before hospital deployment.
- The **PHI guard** (`assertSpineSafe`) is **key-based** — it catches direct-identifier *keys*,
  not a real name hidden in a free-text value; the true masking boundary remains the ehr-adapter edge.
- The **checksum** (FNV-1a) is a non-crypto integrity/drift hash, not a signature.
- Reviews were run adversarially on Slices A / B / C+D (found+fixed ~50 issues); E/F/G reused the
  already-reviewed forms/endpoint patterns and were verified by tests + API + live UI, not a
  dedicated review.
